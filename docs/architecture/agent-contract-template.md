# Agent Contract Template

**Card Version:** 2.0
**Changelog:** §16 Artifact References — added `related_adr` field (now that Stage 8's ADRs exist to point to), and clarified `implementation_reference` as a generic pointer rather than a hardcoded filesystem path (per Stage 7's deferred-naming decision, now resolved). Both items were tracked in the Deferred list since Stage 7; both triggers have now fired.

**Purpose:** Per Card 01 §1, the Contract is a core architectural component on par with Model/Tools/Orchestration — every agent in this ecosystem is instantiated *from* a contract, not designed ad hoc and documented after the fact. This is the field-by-field template. Copy it per-agent into that agent's spec directory (`docs/specs/<project>/<agent-name>-contract.md`) and fill in every field before implementation begins — per Card 07's spec-driven discipline, no agent gets built without an approved, filled-out contract.

**Status:** Template — fill in one copy per agent, do not edit this file per-agent.

**Discipline note:** every field below either declares agent-specific configuration or points to a canonical owner elsewhere — it never redefines architecture that already has a home. If a field tempts you to restate a Risk Tier table, a permission rule, a tool schema, or an evaluation rubric, stop and cite the owning card/section instead.

---

## 1. Identity

| Field | Description |
|---|---|
| **agent_id** | Unique, stable identifier (mirrors the Tool/Skill ID discipline in Card 02 §5 / Card 04 §6 — display names are never authoritative) |
| **agent_name** | Human-readable name |
| **project** | Which of the 3 portfolio projects this agent belongs to, or `core` if shared infrastructure |
| **agent_level** | One of Card 01 §3's 5 levels (0-4). Most agents in this ecosystem should be Level 1-3; Level 4 is explicitly prohibited per Card 01 §8 |
| **contract_owner** | Responsible party for this contract specifically — business/project/system ownership disambiguated by this rename (you, for now) |
| **contract_version** | Versioned independently from the agent's code, using the same stacked-changelog convention as the reference cards (`**Contract Version:**` / `**Changelog:**`, newest entry on top, prior entries preserved below) — a contract change is itself a reviewable event, and its own history should be self-contained without needing git log archaeology to explain *why* a version changed |

## 2. Mission

A single, concise sentence stating what this agent exists to do — not a paragraph, and not a restatement of the project's overall goal. Should be specific enough that "did this agent do its job" has a clear answer.

> *Mission:* [fill in — one sentence]

## 3. Inputs / Outputs

| Field | Description |
|---|---|
| **inputs** | What this agent receives to begin a run — schema or plain description (e.g., "a user query string," "a structured `TransactionRecord`") |
| **outputs** | What this agent produces on a successful run — schema or plain description |
| **input_source_types** | Which `source_type` tags (Card 03 §12) this agent's inputs may legitimately carry — most agents should expect `user_upload`, `tool_output`, or `a2a_message`; flag explicitly if `system`-tagged input is expected, since that carries instruction-level trust |

## 4. Dependencies

What this agent needs already running/available to function, and who else in the ecosystem is connected to it. This is architectural metadata, not implementation detail — it exists to make blast-radius analysis and outage troubleshooting fast instead of requiring an investigation.

| Field | Description |
|---|---|
| **runtime_dependencies** | Major runtime services this agent requires (e.g., Tool Registry, Memory Router, Policy Server) — cite the owning card/section for each (Card 02 §5, Card 03, Card 06 §4), do not redescribe what they do |
| **upstream_agents** | Agents that invoke this agent |
| **downstream_agents** | Agents this agent invokes |

## 5. Tools

| Field | Description |
|---|---|
| **tools_used** | List of Tool IDs (Card 02 §5) this agent is assigned to in the Tool Registry — this contract field and the Registry's `Allowed Agents` field must agree; this contract does not grant access on its own (Card 02 §5's Assignment Authority note — only the Policy Server, Card 06 §4, can actually grant a binding) |
| **risk_tier_exposure** | The highest Risk Tier (Card 06 §3) any of this agent's assigned tools/actions can trigger — informs review depth for this contract itself |

## 6. Memory Access

| Field | Description |
|---|---|
| **memory_types_read** | Which of Card 03 §5's five memory types this agent reads (episodic / semantic / procedural / preference / Structured Records) |
| **memory_types_write** | Which it writes to |
| **memory_contract_reference** | Pointer to the specific Memory Contract (Card 03 §11) governing this agent's read/write scope — this field does not redefine authorization, it points to where authorization already lives |

## 7. State Ownership

Which parts of the shared run `state` (Card 01 §1's definition) this agent is authorized to mutate versus merely read. This matters generally, but is especially load-bearing under LangGraph specifically, where state is directly mutable rather than an append-only event log (Card 03 §4's Implementation Note) — without explicit ownership, two agents touching the same state object is a silent-corruption risk, not just a style question.

| Field | Description |
|---|---|
| **state_owned** | Which state fields/keys this agent may write to |
| **state_read_only** | Which state fields/keys this agent may read but never mutate |

## 8. Execution Constraints

Behavioral boundaries, not performance tuning — these are instantiation-specific limits this contract sets *on top of* Card 03 §9's existing Context Budget Management, not a competing budget mechanism. A Coordinator agent (Card 01 §6) may legitimately need twenty planning iterations; a single-pass Classification agent should need exactly one. Documenting the expectation prevents uncontrolled runaway behavior from looking like a bug only after it happens.

| Field | Description |
|---|---|
| **max_iterations** | Maximum Plan→Act→Observe loop passes (Card 01 §2) before this agent must Stop or Escalate |
| **max_tool_calls** | Maximum tool invocations per run |
| **max_runtime** | Wall-clock budget per run |
| **token_budget** | This agent's specific allocation within Card 03 §9's overall context budget discipline — a number, not a redefinition of the budget mechanism itself |

## 9. Success Criteria

What "Validate" (Card 01 §2) checks against for this agent specifically. State as a checklist of measurable, automatable conditions wherever possible, not prose — prose success criteria are hard to validate consistently and harder to automate.

> *Success criteria:* [fill in as a checklist — e.g.:]
> - [ ] Output schema validates
> - [ ] Evaluator score (§13 below) meets or exceeds target
> - [ ] No Tier 3+ tool call failed
> - [ ] No escalation required

## 10. Stop Conditions

Explicit conditions under which this agent's Decide stage should choose `Stop` (Card 01 §2) rather than continuing to loop. State as a bullet list, not a narrative paragraph — this is operational reference material, reviewed quickly under time pressure.

> *Stop conditions:* [fill in as bullets — e.g.:]
> - 3 consecutive Retry failures on the same tool
> - Mission goal unreachable given available tools
> - Explicit user cancellation
> - `max_iterations` / `max_runtime` (§8) exceeded

## 11. Failure Recovery Strategy

This section maps this agent's encountered Failure Taxonomy categories (Card 05 §6) to a specific Decide outcome (Card 01 §2) — it is a per-agent instantiation decision, not a new retry mechanism layered on top of Card 02 §4's existing per-tool retry policy or Card 01 §2's existing Continue/Retry/Escalate/Stop outcomes. **Recovery outcomes listed here must always resolve to one of Card 01 §2's four existing Decide outcomes — Continue / Retry / Escalate / Stop — never a new execution outcome invented at the contract level.** State which outcome applies to which failure category for this agent specifically:

> *Failure recovery mapping:* [fill in — e.g.:]
> - Memory unavailable (Card 03 §12's degraded-read mode) → Continue without personalization
> - Tool failure on a non-critical tool → Retry once, then Continue without that tool's output
> - Tool failure on a critical-path tool → Escalate
> - Validation failure → Retry with adjusted plan, max 2 attempts, then Escalate

## 12. Escalation Rules

| Field | Description |
|---|---|
| **escalation_triggers** | What causes this agent's Decide stage to choose `Escalate` (Card 01 §2) — e.g., a specific Failure Taxonomy category (Card 05 §6), a Risk Tier threshold, a Safety/Alignment evaluation flag (Card 06 §9) |
| **escalation_target** | Who/what this escalates to — a human reviewer, a higher-tier coordinator agent, or both, per Card 01 §6's HITL pattern |
| **hitl_required_for** | Specific actions this agent may never take without human approval, regardless of confidence — for Projects 1 and 3, this must include any fraud-flag or financial-recommendation output (Card 01's Direct Implications), and anything reaching Tier 4+ by default or Tier 3 under the regulated-project override (Card 06 §3 / §14) |

## 13. Evaluation Metrics

| Field | Description |
|---|---|
| **evaluator_tier** | Which level(s) of Card 05 §4's Evaluator Hierarchy apply to this agent's outputs (Automated / LLM-as-Judge / Agent-as-Judge / HITL) |
| **observability_fields** | Confirm this agent emits the standard fields per Card 05 §5 — do not restate the field list here, point to it (the exact mistake Card 01 §8 made and Stage 3 corrected) |
| **golden_set_reference** | Pointer to this agent's golden-set regression cases (Card 07 §8), if applicable |

## 14. Blast Radius

Not "what happens if this agent fails" in isolation — the more operationally useful question is **what system capability becomes unavailable while this agent is down.** Most agents can simply point to the relevant card-level blast-radius statement rather than restating it.

> *Blast radius:* [fill in, or reference: Card 01 §2 for Harness/Orchestrator-level crash recovery; Card 03 §12 for Memory backend unavailability; Card 02 §5 / Card 04 §6 for Registry unavailability; Card 05 §5 for Evaluation-service unavailability — cite the specific one(s) relevant to this agent's dependencies §4 above, do not restate]

## 15. Assumptions

Explicit architectural assumptions this agent's design relies on. Architects document assumptions; engineers implement against them; operations validates them at runtime. Without an explicit list, failures caused by a violated assumption are disproportionately hard to diagnose, because nobody realized the design depended on it.

> *Assumptions:* [fill in — e.g.:]
> - Tool Registry available (§4)
> - Policy Server operational (Card 06 §4)
> - Required Skills already loaded and Active (Card 04 §6)
> - Input schema validated upstream of this agent (§3)

## 16. Artifact References

Navigation aids pointing to this agent's other concrete artifacts — these are pointers, not new specifications.

| Field | Description |
|---|---|
| **spec_reference** | Pointer to this agent's full spec, if one exists separately from this contract |
| **implementation_reference** | Pointer to the code implementing this agent — intentionally kept as a generic reference (module name, package path, or repo link) rather than a hardcoded filesystem path; exact conventions follow whatever `tech-stack.md` and Phase 1's `core/` structure settle on |
| **test_reference** | Pointer to this agent's test suite |
| **related_adr** | Pointer(s) to any ADR in `docs/governance/adr/` whose decision directly governs this agent's design — e.g., an agent built around a Coordinator pattern cites ADR-001 (Runtime Stack Ordering); a regulated-project agent cites ADR-002 (Policy Server Centralization). Leave blank if no ADR is specifically load-bearing for this agent |

| **approved_by** | Human reviewer |
| **approval_date** | Date |

## Template Maintenance

This template is itself an instruction artifact under Card 06 §15 — any change to its structure (not its per-agent fill-ins) is a Tier 4+ action requiring the same approval discipline as any other high-risk artifact change, and should be logged in the Master Execution Plan's Progress Log.

**Deferred, not included:** Contract Compatibility fields (`minimum_contract_version` / `maximum_supported_version`) were considered during this template's production-readiness review and deferred — not needed at current scale (one developer, no contracts yet versioned independently of each other in production). Revisit if/when Phase 4+ multi-agent cross-version compatibility becomes a live problem; tracked in the Master Execution Plan's deferred-items list, not silently dropped.
